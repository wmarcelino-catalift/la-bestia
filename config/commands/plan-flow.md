# /plan-flow $ARGUMENTS

Like `/flow`, but **stops after Phase 2** and produces a plan document instead of jumping into Develop. Use when:

- You want to share the plan with another human before writing code.
- You're unsure of scope and want a budget estimate first.
- You're at >$20/day spend and want to validate before committing more.

This is the **Plan-Apply ritual** taken to the explicit level — Plan first, separately, then Apply later (with `/flow` resuming or with manual implementation).

## Plan (read-only)

The principal:

1. Runs `bin/flow-estimate.sh "$ARGUMENTS"` first → cost estimate visible to operator.
2. **HALT** — operator confirms estimate is acceptable.
3. Activates `flow-feature` skill but **with `PLAN_MODE=1` flag** in the synthesis instructions:
   - Phase 1 (Discover) runs full
   - Phase 2 (Define) runs full
   - **Stop** before Phase 3 (Develop)
4. Synthesizes both phases into a single `plan.md` artifact.

The plan document has:

```markdown
# Plan: <feature>

> Generated by /plan-flow on <date>. Status: PROPOSED.

## Estimate

- Size: Medium · Touches: ...
- Total: ~$X.XX / ~Y tokens
- Vs Claude solo: -Z% saving

## Phase 1 — Discover findings

- (PR-FAQ summary, top 3 risks, recommended scope)

## Phase 2 — Define decisions

- (ADR draft inline)
- (Locked decisions: stack, storage, schema, API surface)

## What this plan does NOT decide

- (open questions deferred to Develop)

## How to proceed

- `/flow "<same prompt>"` — runs all 4 phases (Phases 1-2 will repeat or you can ask to skip)
- Manual implementation following the plan
- `/cto-review plan.md` — adversarial review of this plan before proceeding
```

## Apply (gated)

The plan is written to `<repo>/plans/<NNNN>-<slug>.md` only after operator confirms.

If operator says `proceed` → optionally launches `/flow` with Phase 1 cached (or skipped if Plan was thorough).

## Cost

- Plan-only run: ~50% of full `/flow` cost (no Phase 3-4)
- Full /flow after plan accepted: ~50% additional (Phase 3-4 of original)
- Total: roughly equal to full /flow, but with a CHECKPOINT in the middle

## When NOT to use

- Trivial change → just edit
- Small (<1 day) feature → use `/flow` directly, plan ritual is overhead
- Bug fix → `/deep-debug` is the right tool

## Idempotency

Re-running `/plan-flow` with the same prompt and same hot-context produces the same plan. The output file is overwritten only with operator confirmation.

## Chains

- `bin/flow-estimate.sh` — runs first to surface budget
- `flow-feature` skill — invoked with PLAN_MODE flag
- `@cto-review` — recommended next step to validate the plan before committing

## Difference from `/flow`

| Behavior                 | `/flow`            | `/plan-flow`                       |
| ------------------------ | ------------------ | ---------------------------------- |
| Runs Phase 1 (Discover)  | yes                | yes                                |
| Runs Phase 2 (Define)    | yes                | yes                                |
| Runs Phase 3 (Develop)   | yes                | NO — halts here                    |
| Runs Phase 4 (Deliver)   | yes                | NO                                 |
| Output                   | code + ADR + tests | `plans/<NNNN>.md` artifact         |
| Cost                     | full               | ~50%                               |
| Operator decision points | 3 HALTs            | 1 estimate-confirm + 1 plan-accept |
