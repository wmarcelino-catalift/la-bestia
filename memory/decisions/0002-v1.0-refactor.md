# ADR-0002 — La Bestia v1.0 refactor (drop Obsidian, drop UI, add foundation)

**Status:** accepted
**Date:** 2026-05-03
**Deciders:** wmarcelino-catalift
**Tags:** architecture | breaking-change | governance
**Supersedes:** ADR-0001

## Context

By v0.4 the harness had drifted into three problems:

1. **Hidden dependencies on Obsidian.** `~/Obsidian/claude-brain/` was inlined into `live-update.sh`, `inject-context.sh`, `curate-hot.sh`, the constitution, and `install.sh` (which auto-registered an Obsidian MCP). New operators had to install Obsidian or rip out references to onboard.
2. **UI-shaped artifacts in a non-UI harness.** `dashboard.sh`, `flow-diagram.sh`, the `graphify` skill, and `live-update.sh` (writing emoji markdown to a vault for some external viewer) were visualization code we did not own a viewer for. They added surface area without value.
3. **Zero foundation.** No tests for the hooks. No CI. No schemas to reject malformed agents. No CHANGELOG. No SECURITY.md. No CONTRIBUTING. No evals. No MCP guidance. v1.0-claim with v0.1-engineering.

Comparator analysis (vs `wshobson/agents`, `claude-code-pm`, emergent.sh) showed the gap: not in agent count, but in foundation.

## Options considered

1. **Layer foundation on top of v0.4** — keep Obsidian + UI, add tests/CI/schemas around them.
2. **Delete vault and UI in same release as foundation** — one breaking version, clean baseline.
3. **Two-phase: delete in v0.5, foundation in v1.0** — slower, but smaller blast radius per release.

## Tradeoffs

| Dimension                    | Weight | Opt 1 (layer)                    | Opt 2 (single major) | Opt 3 (two-phase)    |
| ---------------------------- | ------ | -------------------------------- | -------------------- | -------------------- |
| Test/CI surface to design    | 3      | larger (vault paths, UI scripts) | minimal              | minimal then minimal |
| Operator migration burden    | 2      | none                             | one                  | two                  |
| Time to "looks professional" | 3      | never (drag)                     | now                  | +1 sprint            |
| Reversibility                | 1      | high                             | low                  | medium               |

## Decision

**Option 2: single major release.**

v1.0 ships:

- Removed: `dashboard.sh`, `flow-diagram.sh`, `live-update.sh`, `curate-hot.sh`, `skills/graphify/`, all Obsidian references in code and docs.
- Added: `ARCHITECTURE.md`, `CHANGELOG.md`, `CONTRIBUTING.md`, `SECURITY.md`, `schemas/` (4 schemas), `tests/` (3 bats files + runner), `.github/workflows/` (ci + evals), `evals/` (framework + 1 example), `mcp/` (templates, no auto-install), `memory/templates/` (ADR + pattern), component `_TEMPLATE.md` files.
- Refactored: `inject-context.sh`, `track-agent.sh`, `log-session.sh`, `statusline.sh`, `verify.sh`, `sync-to-bestia.sh`, `install.sh`, `settings.example.json`, `config/CLAUDE.md`, `README.md`.
- Memory: only two layers — repo `memory/{hot-context,decisions/,patterns/}` and global `~/.claude/agent-memory/<agent>/`. No external vault.
- MCP: zero servers wired by default. Templates in `mcp/README.md` for operator opt-in.

## Consequences

**Positive**

- Operator can `git clone && bash install.sh global` without installing Obsidian.
- CI runs on every PR with hard gates (shellcheck, bats, schema validation) — no more "compile in production".
- New contributors can copy `_TEMPLATE.md` and follow `CONTRIBUTING.md` to add an agent in < 30min.
- The harness fits the "boring tech wins" principle (#9): bash + jq + bats + GitHub Actions, no bespoke runtime.

**Negative / accepted costs**

- v0.4 → v1.0 is a breaking change. Migration documented in CHANGELOG.
- Operators relying on `live-update.sh` lose the live session viewer. This was a single-operator (wmarcelino) feature; acceptable cost.
- Eval coverage is 1/18 agents at v1.0 ship. Backfill is the next sprint.

**Neutral / to monitor**

- p99 hook latency. Budget: 200ms. Test harness for this is TODO (`tests/perf/`).
- Operator install friction. Watch for issues filed within first month of v1.0 publication.

## Failure modes

1. **Failure:** CI on Windows Git Bash differs from Linux runners → contributors with Windows-only setups can't merge.
   **Detect:** Issue filed by Windows contributor.
   **Mitigate:** Add Windows runner to `ci.yml` matrix when first such issue appears.

2. **Failure:** `block-secrets.sh` regex blocks a legitimate write a contributor needs.
   **Detect:** PR comment "the hook blocks my legit X".
   **Mitigate:** Add bats test capturing the case, refine regex, ship as patch release.

3. **Failure:** Schema becomes too strict, blocks innovation in agent prompts.
   **Detect:** New-agent PRs failing schema repeatedly without agent-frontmatter bugs.
   **Mitigate:** Relax schema, version-bump, document in CHANGELOG.

4. **Failure:** Operators paste secrets into agent-memory `MEMORY.md` files (block-secrets only fires on Write/Edit, not retroactively).
   **Detect:** Periodic gitleaks scan in CI (TODO).
   **Mitigate:** Add agent-memory pre-commit scan; document hardening.

## Reversibility

- [x] **One-way door.** Restoring vault/UI artifacts post-v1.0 would re-introduce the dependencies we just removed. We accept this.

Off-ramp if v1.0 was wrong: tag v0.5 as "v0.4 + foundation only" and revert vault deletions. Cost: rebuilding tests against vault paths. ~2 days of work.
